Investigation

My search takes me to here:
https://github.com/nodejs/node/blob/main/lib/internal/process/promises.js#L166

When the Promise was rejected but not handled (not passed a then or catch or finally phrase), it gets added to a weak map maybeUnhandledPromises and array `` by node. The warned field is currently false, and set to true in the next tick by processPromiseRejections.

But later I chain to this promise a then or catch or finally that attempts to handle the rejection, the promiseRejectHandler will try to handle the rejection, the type will be kPromiseHandlerAddedAfterReject - quite self explanatory. handledRejection gets called. handleRejection finds out that this promise is still in maybeUnhandledPromises with warned set to true, therefore it fires a PromiseRejectionHandledWarning.

This only warns once because once this rejection is handled, it gets removed from maybeUnhandledPromises by handledRejection:
