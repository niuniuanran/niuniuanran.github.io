IAM policies

Managed Policies

A policy managed by AWS. cannot be edited.
labeled with an orange box.

Customer Managed Policies

created by user.
editable.
no symbol beside them

Inline Policies

Directly attached to the user/group/role.

Policy Structure

  • Version
  • Statement
    • Sid(optional, labelling)
    • Effect (Allow/Deny)
    • Principle (account/user/role/federated user to which you would like to allow/deny access. You cannot use the Principal element in an IAM identity-based policy. You can use it in the trust policies for IAM roles and in resource-based policies. )
    • Action (eg. iam:CreateServiceLinkedRole)
    • Resource (the resource to which the principle applies. eg. arn:aws:s3:::my-bucket)
    • Condition (optional: circumstances under which the policy grants permission)

An IAM role is both an identity and a resource that supports resource-based policies.

Resource-based policies – Attach inline policies to resources. The most common examples of resource-based policies are Amazon S3 bucket policies and IAM role trust policies. Resource-based policies grant permissions to the principal that is specified in the policy. Principals can be in the same account as the resource or in other accounts.

Programmatic Access Keys

allow user to interact with AWS programmatically with AWS CLI or AWS SDK
You are allowed 2 access keys per user.

If your root account has access keys, those should be removed. The root account most likely will be used just for console access and not AWS CLI or SDK access.

Access keys are shown once when they were created. Have to be deleted+recreated if lost.

MFA

MFA cannot be directly enforced by root account administrator for each user. The user has to turn on MFA themselves.
But root account administrator could create a policy requiring MFA to access certain resources.
