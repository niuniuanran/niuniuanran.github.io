    Class and its name

    Naming and how class interacts with other parts of the programme are two different things, both having impacts on the comprehensibility of the programme.

    • Name should be able to capture what interactions it will have with other parts of the programme.

    Object-oriented design

    What is object-oriented programming?

    • History
      • Many discussion talk about programming language features.
      • First OO language: Simula - not very widely known
      • Smalltalk: at the Learning Research Group (LRG) of Xerox PARC by Alan Kay and others during the 1970s.
      • C++ by Bjarne Stroustrup, 1985
    • It is about how you organise your code, not about specific language.
    • Language features: what features does a language need to have to be considered “OOP”?
      • objects
      • encapsulation (what is encapsulated? -> objects)
      • class (not with prototype-based inheritance: Self, Javascript)
      • inheritance (not that important for OOP. useful but not important)
      • types (Smalltalk, python: no static type checking)
      • generic types (Smalltalk, Java before generics)
      • polymorphism (runtime decision to decide which code to execute. aka virtual function call, dynamic dispatch, message send)

    Key features for OOP

    working in a object-oriented language is neither a necessary nor sufficient condition ofr doing OOP.

    The two key features for OOP: objects and polymorphism: Both are Runtime concepts.

    OOP definition

    An object-oriented program is one that when it executes, creates objects and sends messages to each other.

    “The most important aspect of OOP is the creation of a universe of largely autonomous interacting agents.”

    An object-oriented design is one that describes OOP.

    Consequences

    • Encapsulation is a consequence of interaction between objects only being by message sent.
    • Inheritance is irrelevant to OOP - is relevant to design quality.

    Objects

    Traditional description:
    - Identity
    - State
    - Behaviour

    Finding good objects

    • Reflection of real world? -> Good start point, but there are some good objects that do not exist in the real world.
      • Connections with Comprehensibility: If we know the relevant part of the “real world” (in our KB) then objects that correspond to it will be easier to understand.
      • (Provided that the objects really behave like the entities in the real world that they are supposed to correspond to. Conflicting behaviours will reduce efficiency of comprehensibility).
    • All advice points to choosing objects that “make sense” with respect to the problem being solved.

    Martin on classes

    Organisation

    “Standard” vertical order

    • public static final fields (constants)
    • other static fields
    • instance fields
    • constructors
    • public methods
      • what order?
    • non public methods (but those called by public methods close to the methods they’re called from)
      • not always possible -> Try to order method with a logical relation closer together.
      • what about protected versus private?

    Questions:

    • If a private method is called by three different public methods, which should the private method be close to?
    • Getters and Setters?
      • Can be treated the same as other public/private methods.
    • The important thing: people follow the same practice so that it is easier for people to locate things.

    Encapsulation: no just Visibility

    Visibility

    Fields and “utility functions” usually private. Protected is acceptable sometimes.
    There is seldom a reason to have a public [field].

    Abstraction vs details

    Direct access to representation details of abstraction from outside the abstraction is a bad idea: don’e expose the implementation details.

    If you do make fields public, you create the possibility that the clients can use them in ways different than what the abstractions are supposed to do.

    • The design question: “What methods are needed to support the abstraction?”
    • Sometimes exposing getters about the choice of representation but does not support the abstraction of the class is supposed to represent “bad encapsulation”.

    Small

    We know the less an object represents a concept in the context schema/design schema, the less comprehensible the design.

    Observation: the smaller the class, the more likely it will represent a specific concept in the context/design schema : “Small” is the effect.

    Single Responsibility Principle (SRP)

    One and only one reason to change (only one concept change in the schema will cause the class to change).

    Classes that only have one responsibility or only one reason to change are likely to be small.

    Cohesion

    Essentially the same as single responsibility

    Key points

    • Match concepts in the context and design schema
    • Encapsulation: an object’s exposed behaviours <=> the concept it represents
    • objects that are a good match with a specific concepts: SRP, Cohesive
    • Classes with good objects tend to be “small” (for some definition of small). Small is the effect/result of good match with a concept.
