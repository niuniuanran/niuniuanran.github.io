Implementation 1

Speculative Generality

I feel that the author worked really hard to give the space for future changeability, but goes too far by assuming what changes will be happening. I see myself through it.
For example, the author made methods in the Container class that are created to be overwritten. The superclass’s implementation do not really make sense and are just “dummy” implementation. However, only one subclass needs to overwrite the method. Why not just move it down to the subclass itself, as the House or Store‘s own behaviour? If in the future, we have more subclasses of Container that will also need to implement methods like removeSeed, it is not a hard change to move the method up.

The interfaces look more luring as it does not add misleading concrete method implementations. But still I think it is a waste to make assumptions on what methods will need to be reused and extract an interface so early. It is always easy to extract an interface based on real need. When we only know of one concrete class, what’s the point?

The long methods like move is bit harder to wade into. Will come back to it.

Assignment Requirements

Code Review

  • You are to review two Java implementations of Kalah, and comment on the maintainability of each implementation.
  • You should think carefully about the type of comments you could make and how these comments could help someone else to improve the maintainability of the given implementations.
  • You should also post an overall comment in each pull request summarising the comments you made.

    Report

  • You must submit a short report comparing your implementation of Kalah from Assignment 1 with the two given implementations.
  • Your report should consists of 2 paragraphs (one for each implementation), explaining why your implementation is better or worse in terms of maintainability than each implementation.
  • In your report, you should also rate each implementation (including yours) on a scale from 1 (I never want to work on this code) to 5 (The code is very maintainable).
  • You must use the IEEE conference template, using A4 page format, 10pt font for the body, and conference style. You must submit a PDF file based on this template. The title can be simple (e.g. “Assignment 2”). It should include your username in the author field (as well as your name). You do not need an abstract, introduction/conclusions, etc.
