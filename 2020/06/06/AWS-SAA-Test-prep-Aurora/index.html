Fully Managed Postgres or MySQL compatible database, designed by default to scale and fine-tuned to be really fast.

Introduction to Aurora

Combines the speed and availability of high-end databases with the simplicity and cost-effectiveness of open-source databases
Aurora can run either MySQL or Postgres compatible engines.

Aurora MySQL: 5* better performance than traditional MySQL
Aurora Postgres is 3* better performance than traditional Postgres.

1/10th the costs of other solutions offering similar performance and availability.

Scaling with Aurora

Storage: autoscaling, Start with 10GB. Scale up in 10GB increments up to 64TB.
Computing: can scale all the way up to 32vCPUs and 244 GB of memory.

Availability with Aurora

A minimum of 3 AZs, each containing 2 copies of your data at all times: 6 copies
Lose up to 2 copies of your data without affecting write availability
Lose up to 3 without losing **read availability

Fault Tolerance and Durability

Aurora backup and failover is handled automatically.
Snapshots of data can be shared with other AWS accounts.
Storage is self-healing, in that data blocks and disks are continuously scanned for errors and repaired automatically.

Aurora - Replicas

Two types of replicas available:

Amazon Aurora Replicas

Up to 15 replicas.
Asychronous replica in milliseconds
Performance impact on primary: Low
AUtomated failover: Yes
Act as failover target: Yes, no data loss

Support for user-defined replication delay: No
Support for different data/schema as primary: No

MySQL Read Replicas

Up to 5 replicas - same as other RDS read replicas
Asychronous replica in seconds
Performance impact on primary: High
Automated failover: no
Act as failover target: Yes, potentially minutes of data loss

Support for user-defined replication delay: Yes
Support for different data/schema as primary: Yes

Aurora Global Database

An aurora database can span multiple regions via Aurora Global Database

Aurora Serverless

Ideal for new projects or projects with infrequent database usage.
Aurora Serverless is Aurora except that the database will automatically start up, shut down, and scale capacity up/down based on your application’s needs.
You pay for database storage, database capacity and I/O your application consumes while it is active.

You have two options of setting up your Aurora:

  • One writer and multiple readers: general purpose option for most workloads
  • Serverless: when you have intermittent or unpredictable workloads

You can’t give an Aurora Serverless DB cluster a public IP address.
You can access an Aurora Serverless DB cluster only from within a virtual private cloud (VPC) based on the Amazon VPC service.

Each Aurora Serverless DB cluster requires two AWS PrivateLink endpoints.
