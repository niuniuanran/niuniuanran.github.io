CompSci701 Assignment2 Reflection

For assignment2, we conducted code review on two implementations of Kalah, from the perspective of comprehensibility. The two important takeaways are:

Being aware of how useful my comments are:

  • Provide valid alternatives.
  • I am not re-implementing for the author. The code review could just include some comments pointing the author to the right direction.
  • Be concise, but polite.

Creating meaningful classes/interfaces:

  • No need to extract everything that “might” change in the future so early. However, decoupling components by letting each component encapsulate their implementation details will be very helpful.
  • Don’t choose not to create a class because it is “simple”. If it is a good match with a context concept, go for it. For example, the Pit, Store and House classes are such a good match with the context.

Timing of refactoring?

After looking at implementation one, I have more thoughts about when to refactor.
Before I think that big refactoring should happen immediately after I have implemented the code. I thought I should look at the code and think, what will need in the future, and how to prepare for them.
Now I think big refactoring might be more efficient before I need to extend or change.
Of course, I still should review and improve my code continuously, giving a better name and better structure etc., but major refactorings, such as extracting a new inheritance hierarchy, will be much more efficient when I have a realistic direction of how the code needs to be changed or extended.
