Single Responsibility Principle

A class should have one, and only one, reason to change.

Actionable

  1. One reason to change.
  2. Group things that change for the same reason together.

Alterability

Detailed analysis here, reasoning based on the actionable â€œone reason to changeâ€.

Open-Closed Principle

  • You should be able to extend a classâ€™ behaviour without modifying it.

Actionable

  • Achieved through judicious use of polymorphism.
  • Lead to design with abstract entities that are made concrete through:
    1. Extending the abstract entity, or
    2. Supplying concrete implementations as parameters (Dependency Injection)
  • When designing something, think about what variations are needed for the Context Schema

Alterability

Reduce classes being changed - Reducing C.

Related Design Principles

  1. Dependency Injection: change which implementation of an abstract service is used
  2. Strategy Pattern: change how a given behaviour in a context is conducted
  3. Template Pattern: change the sub-step of a procedure that belong to an abstract entity
  4. Factory Pattern: instantiate different implementations of an abstract entity

Liskov Substitution Principle

Where a parent class is used, we should be able to replace the parent class with any of its child classes, without breaking the programme.

Actionable

LSP Provides actionable criteria to determine whether â€œinheritanceâ€ has been used â€œproperlyâ€

Alterability

LSP is about designs that allow changes in behaviour through substitution without requiring the code using the parent (the context) to change -> reducing C.

Related Design Patterns

Any design pattern that relies on inheritance (pretty much all of them) is based on LSP

  • Command: execute()
  • Observer: update()

Dependency Inversion Principle

Depend on abstractions, not on concretions:

  • High-level modules should not depend on low-level modules. They should both depend on abstraction.
  • Abstraction should not depend on details. Details should depend on abstraction.

Alterability

Reduce dependency on concrete implementation
=> Reduce the need to change existing context when the concrete types change
=> Reduce C

Actionable

Look for opportunities to use abstractions rather than concretations

Design Patterns

  • Dependency Injection: the client depends on the abstract service
  • Strategy Pattern: Context depends on abstract strategy
  • Decorator Pattern: Context depends on abstract component, Decorator depends on abstract component too.

Cohesion

  • The extent to which its individual components are needed to perform the same task
  • How tightly bound or related a moduleâ€™s internal elements are to one another.

Coupling

  • The degree of interdependence between modules
  • A measure of the strength of interconnection

Relationship with Design Principles

ISP

The more one class knows about another class, the stronger the coupling.

DIP

Depending on a concrete thing has higher coupling than depending on an abstract thing.

LSP

  • The context code expects objects from a particular class A
  • Objects from B can be substituted if B is a descendant of A and LSP is followed
  • Therefore the coupling between context and the other classes is not particularly strong: if a concrete A is not available, the context still works.

SRP

  • Classes that have more than one responsibility likely have more classes that they depend on (for each responsibility) than those that follow SRP (high FAN-OUT)
  • Classes that have more than one responsibility likely have more classes that depend on them (for each responsibility) than those that follow SRP (high FAN-IN)

Definition

Testability â€” degree of effectiveness and efficiency with which test criteria can be established for a system, product or component and tests can be performed to determine whether those criteria have been met.

Trade-Off

  • Encapsulation helps comprehensibility and alterability, but hinders testability.
  • For testability you want:
    • Controllability
    • Observability
    • Avoid breaking encapsulation if possible (or reduce the effect as much as possible).
  • In particular, dependency injection gives us control as to what the state is.

Good Test Cases

  • Fast
  • Independent
  • Simple
  • Repeatable
