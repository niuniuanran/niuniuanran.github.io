🐾🐾🐾 Lecture Review: Classes 🐾🐾🐾

OOD

  • “An object-oriented program is one that when it executes creates objects that send messages to each other.”
  • Objects + Polymorphism: Run-time concepts
  • Static members are not part of an object-oriented design
  • Classes are describing and creating objects

Finding Good Objects

  • Good object helps you make a good guess as to what it does, without referring to external representation.
  • Code should match our expectation (context schema)
  • One aspect of “match”: number of objects created from the class <-> what we expect from the context schema.

Comprehensible Classes

Classes should represent concepts from the context schema or design schema.

Organisation (Ordering)

  • Ordering should match what the user expect, so they will efficiently find what they need to access from the code base.
  • Methods of higher abstraction level then methods of lower level/more specific implementation details.

Encapsulation

  • Objects (and classes) represent concepts that appear in the context schema.
  • Implementation details should not be exposed.
  • The question to ask: what methods are needed to support the abstraction

SRP -> Small

  • Have only one reason to change
  • Small: Do not use “it’s too simple” as an excuse to not create a class!

Cohesion

  • The less tightly bound the internal elements, the more disparate the parts to the module, the harder it is to understand.

🐾🐾🐾 Lecture Review: Alterability 🐾🐾🐾

Defining Alterability

Degree to which a product or system can be effectively and efficiently changed without introducing defects or degrading existing product quality.

  • NOT counting costs associated with comprehension.
  • Assume the mental model is “complete” when defining alterability

Assessing Alterability

Change Case to Describe Changes

  • Likelihood
  • Impact
    1. against the requirements/behaviour
    2. What proportion of the original “cost” of the product are you prepared to pay for this change case?

C/N (Change Ratio) measurement

We want to reduce C/N.

🐾🐾🐾 Lecture Review: Design Pattern 🐾🐾🐾

Defining Design Patterns

  • general solution
  • solution: not an alternative
  • commonly problem
  • given context

Design Patterns and Alterability

By definition does not support alterability, but many problems are effectively “how to support change” -> Design patterns tend to have some impacts on alterability.

More details in another journal (Design Patterns in One Image and One Sentence).
