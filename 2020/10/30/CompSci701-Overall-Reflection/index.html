CompSci701 Overall Reflection

CompSci701 is honestly my favourite course up to now. And interestingly, it has been a favourite course of almost all the potential graduate employers that I interviewed with.

I am really grateful for the opportunity to take the course, and grateful for Ewan and Yu-Cheng‚Äôs generality in running this course and sharing your insights and experiences.

1. My Favourite Aspects

1.1 Examples close to real work

The part I enjoyed the most in CompSci701 was the design examples closer to real work. I learnt about design patterns and design principles from the over-simplified examples of ducks, diners and pizza shops. I thought I understood them but this course showed me how different it was to apply design patterns on ducks and apply design patterns on an actual project.

CompSci701 really helped me to fill the gap between ducks and industry-relevant design thinking. For me the difficulty and complexity of examples felt quite comfortable throughout the course. Especially, we had links to frameworks actively in use among developers, like Spring Boot and JUnit.

I also like it that we were exploring Swing quite a lot - The family of Swing interfaces/classes were far~ more vivid examples of diverse design than the duck family.

I am really grateful that instead of reusing the examples of each pattern/principle that every tutorial website uses, you came up with fresh examples that gave new insights. Before these new examples from CompSci701, I honestly think Command Pattern has to involve a list of history commands and ‚Äúundo‚Äù, as all the examples I saw had this.

2. The Part I did not enjoy so much

L1.03: Meaningful Names

I did not quite enjoy this specific discussion because I felt that the naming patterns discussed went too far to tell us much about improving our own practices.

Maybe it was me lacking imagination about how far a developer can go, but who, working on a serious project, would write ( superman + starship ) / god;?

I think it would be nice to discuss ‚Äúnot so good‚Äù practices instead of these ‚Äúrandom‚Äù practices.

Maybe something like assignment 2 - we can imagine a real person writing it, and we might write something like it too, and therefore the discussion might better refine our design practices.

3. What would be nice to have

Maintainable Open-Source project example

I believe there will be many nice open-source projects on github that will be great learning resources. We might be able to take advantage of that in the course, examining how maintainable there code base is, and even attempting to contribute to their code base in a maintainable way?

I have always wanted to start contributing to open-source projects but found it hard to get started as I get hesitant about my practices. It would be really great to kick-start a open-source journey from a course like this.
