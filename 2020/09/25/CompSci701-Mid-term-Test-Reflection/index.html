CompSci701 Mid-term Test Reflection

What was tested

We had the mid-term test, which focused on Comprehensibility of software. The three important parts being tested were:

  1. Program Comprehension Model, and how each component of the model is related with our comprehension process;
  2. Common anti-patterns (smells) and how they impact the efficiency and effectiveness of comprehension;
  3. Maintainable naming, functions and classes.

Takeaways

The parts that I felt more confident in after the test include:

  1. Efficiency and effectiveness of comprehension
  2. Understanding the components of the comprehension model

The points that I felt most unsure about was:

  1. Useful v.s. Not useful code review
  2. Some refined differentiation between components of the comprehension model

Question to Takeaway

Question 2 (Comprehension Model Components)

Suppose you were trying to understand some code and you come across the following statement:

int result = obj.someMethod();

In order to understand this statement you need to know what kinds of values the variable obj has. Which of the following parts of the comprehension model discussed in class is the best choice for where would this information be recorded?

A. Mental Model

The definition for Mental Model is “The developer’s current understanding of the program under consideration, e.g. known parts and current hypotheses”. In this case, the current understanding would include what kinds of values the variable obj has, if the reader happens to remember what obj was when she saw its definition earlier.

B. Assimilation Process

Assimilation Process is not where we store stuff.

Question 11 (Classes)

Suppose you had to develop an App for managing contacts. A contact consists of a person’s name or an organisation’s name, any number of street addresses, any number of postal addresses, any number of email addresses, and any number of social media accounts. Possible classes you might create include: Contact, Name, StreetAddress, PostalAddress, EmailAddress. However more classes are possible. Suggest three (3) more classes, choosing names that make it clear how they would apply to the App.

Remember the advice given at the end of Lecture 6.1 - do not use “It’s too simple” as a reason to not create a class. Other possible classes include: Person, ContactType (person, organisation) GivenName and FamilyName (for names of people), City (including postal code), SocialMediaType (Twitter, Facebook, etc). Other possibilities include: EmailAddressList, OtherNamesList, StreetAddressList, PostalAddressList. Other classes were accepted if they were obviously relevant, or the relevance was explained. (e.g. PhoneNumber) It might be tempting to suggest a parent class for the different kinds of addresses, but there is not much commonality between an email address and a street address, so this is not as good an answer as the others.

In this question, I did not think of classes to represent simple concepts, such as City; instead, I expanded to the MVC and presented two UI component classes: ContactInputPanel and ContactListPanel.
